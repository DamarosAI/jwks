Trust doctrine

The model is not the system of record.

This document describes how Damaros governs AI use, clinical data handling, eligibility execution, and audit reconstruction for decentralized trial-access deployments. It is written for security, compliance, clinical operations, and integration reviewers evaluating site, sponsor, or network diligence.

00_SCOPE

Document scope

Damaros is the operating system for decentralized clinical trials. Community clinics run as governed trial-access nodes: site-approved evidence in, deterministic screening, human review, replayable referral packet out.

This page states platform-level trust boundaries. Contract-specific retention, subprocessors, incident response SLAs, and jurisdiction terms live in the executed BAA or DPA for each deployment. Technical wire-level integration detail lives in the FHIR integration reference.

01_AI_SURFACE

Permitted and prohibited AI surface

Permitted upstream

  • Non-PHI protocol text, amendments, and sponsor documents.
  • Criterion structuring, ambiguity detection, and mapping suggestions via Trident.
  • Governance narration and operator support outside verdict issuance.
  • Policy-bound tasks with provenance and review hooks captured by Luna.

Prohibited on patient path

  • PHI-bearing patient evidence parsing on the screening path.
  • Eligibility verdict issuance, adjudication, or replay reconstruction by model output.
  • Audit source of truth generation from generative answers.
  • Any model as system of record for patient-path decisions.
Hard boundary. PHI never touches an LLM on the eligibility path. Optional external requests never serialize patient identifiers, MRNs, cohort-linked clinical fields, or free-text narrative into any AI-assisted path.
ENFORCEMENTScreening pipeline accepts normalized protocol-scoped facts only. Luna stamps AI origin and policy scope before downstream use. Default egress denies patient-identifiable transmission to third-party assistance endpoints for screening.
FAILURE_MODEIf required evidence is missing or ambiguous, output is REVIEW or FAIL with rationale. The system does not infer eligibility from a model guess.
02_EXECUTION_SPINE

Protocol to referral, one governed chain

Every clinic node runs the same execution spine sponsors deploy. Versioned protocol logic binds to site-approved evidence, screens deterministically, routes REVIEW work, captures human adjudication, and ships a replayable referral packet.

01Protocol lock

Trident compiles language into versioned, executable eligibility structure. Amendments create a new protocol version; prior runs remain tied to the version that executed.

02Evidence admission

FHIR cohort sync and approved sources enter with lineage, retrieval timestamps, and resource references. Only contract-mapped fields persist for evaluation.

03Deterministic screening

Same protocol version, same evidence snapshot, same evaluation_as_of: same criterion outcomes. PASS, REVIEW, or FAIL per criterion with attached evidence references.

04Replay

Protocol version, evidence snapshot, engine run, reviewers, adjudication state, and referral packet hash reconstruct under audit.

IMMUTABILITYScreening runs and locked protocol versions are append-only once written. Corrections occur through new runs or explicit adjudication records, not silent overwrite.
VERDICT_SEMANTICSPASS: criterion satisfied on admitted evidence. REVIEW: missing, conflicting, or human-judgment-required evidence. FAIL: criterion not satisfied on admitted evidence.
03_HUMAN_REVIEW

Human review before referral

Deterministic screening surfaces REVIEW worklists for coordinators and investigators. Human adjudication captures who reviewed, what changed, and when. Clinician confirmation precedes referral packet issuance.

  • REVIEW queues group patients by blocker type: missing labs, ambiguous criteria, stale evidence, protocol amendment drift.
  • Adjudication records attach to the replay bundle. Sponsor views see de-identified load and completion signals, not raw chart content.
  • Referral readiness requires explicit site confirmation. The platform does not auto-refer from engine output alone.

05_REPLAY_BUNDLE

What replay reconstructs

A replay bundle is the institution-defensible artifact sponsors, CROs, QA, and auditors use to reconstruct a decision. It is not a model transcript.

  • Locked protocol version ID and engine version.
  • Evidence snapshot references: FHIR resource IDs, retrieval timestamps, cohort membership at sync time.
  • Criterion-level outcomes with PASS, REVIEW, or FAIL, rationale, and evidence pointers.
  • Human adjudication history: reviewer identity, timestamps, and state transitions.
  • Referral packet status, confirmation actor, and integrity hash.
  • evaluation_as_of timestamp fixing the temporal basis for time-sensitive criteria.
Citation vs replay. A citation shows what a model read. Replay reconstructs the decision the institution made under locked rules and admitted evidence.
06_TRIDENT_AND_LUNA

AI substrate with enforced separation

  • Trident compiles protocol language into executable eligibility structure. It sharpens with every protocol release and site run. It never issues a patient verdict.
  • Luna records AI origin, policy boundaries, provenance, and execution constraints. AI drafts language-heavy work upstream. Luna stamps what crossed the boundary and what did not.
  • Trident output is versioned and reviewable before lock. Locked protocol logic is what the deterministic engine executes.
  • Luna audit entries cover optional AI-assisted tasks outside screening: drafts, suggestions, operator support. They do not replace criterion outcomes.

09_CATEGORY

Discovery vs governed access

Others validate AI-assisted trial discovery. Damaros builds the governed execution layer sponsors, CROs, and research networks need to operationalize and defend those decisions across community clinics. Discovery suggests. Governed execution proves.

10_VALIDATION

Validation dimensions and cutover posture

Validation measures execution integrity, not clinical efficacy. The same Protocol → Evidence → Screening → Replay chain runs in validation and production so retrospective evidence carries forward to cutover.

  • Criterion-level agreement, false PASS, false FAIL against adjudicated reference sets.
  • REVIEW burden and missing-evidence drivers by criterion and site.
  • Evidence-lineage completeness and replay completeness.
  • Adjudication deltas between runs for fixed protocol, cohort, and evaluation_as_of.
  • Run reproducibility: re-execution on the same snapshot produces identical criterion outcomes.
Production cutover is a credential and environment swap, not a workflow rebuild. Sandbox and production credentials, JWKS URLs, and signing keys remain isolated.
