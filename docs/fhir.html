Epic / FHIR

Enterprise integration reference

FHIR is the enterprise integration spine for Damaros: read-only Backend Services access, site-approved cohort sync, a secure JWT and JWKS posture, and deterministic execution on governed evidence. The same execution chain runs validation and production, so validation evidence carries forward to cutover without rebuilding workflow logic.

00_SUBMISSION_NARRATIVE

Research execution on approved cohorts, not ad hoc analytics.

Damaros is clinical-trial execution infrastructure for site-approved cohorts. Site-approved FHIR access binds locked protocol logic to a defined evidence cohort, evaluates eligibility deterministically, and preserves criterion-level evidence with replay lineage on one audit path.

Deterministic core. Versioned protocol logic plus normalized clinical facts from REST and optional bulk export feed a single eligibility engine. Outputs are PASS, REVIEW, or FAIL at criterion granularity with evidence and sync provenance. No generative verdict on the screening path.

Trident is trial knowledge that sharpens with every protocol release and site run, compounding into executable logic. Luna is the governance harness where AI-assisted work becomes traceable, reviewable, and safe.

No shadow EHR. Damaros is not a secondary warehouse. Ingest ends in protocol-scoped screening artifacts. Epic remains system of record for longitudinal care.
01_PLATFORM_OVERVIEW

Platform overview

The operating chain is Protocol → Evidence → Screening → Replay. Versioned protocol compiles to rules; FHIR sync materializes the cohort as governed evidence; screening records PASS, REVIEW, or FAIL per criterion; replay preserves the protocol version, evidence, and execution context for coordinators, QA, and audit.

Production uses SMART on FHIR Backend Services: a signed JWT exchanged for an access token (client_credentials + private_key_jwt, RFC 7523). Read-only for evidence cohort and screening ingest. Tokens stay server-side in process memory, never in browsers.

02C_OAUTH_SCOPE

Initial requested scopes

An OAuth floor of eight system/*.read scopes covers the per-patient eligibility bundle plus cohort membership via GET Group/{id}. No write scopes on first deployment. The connector never calls FHIR Binary; DocumentReference ingest keeps metadata only.

  • system/Patient.readPatient resource for cohort binding and demographics used in deterministic rules
  • system/Group.readResolve Epic cohort membership for approved trial evidence lists
  • system/Observation.readLaboratory observations per Epic-tuned searches, not a full-chart extract
  • system/Condition.readActive and problem-list context for inclusion or exclusion criteria
  • system/MedicationRequest.readOutpatient medication orders for protocol-relevant criteria
  • system/Procedure.readHistorical procedures referenced by criteria
  • system/AllergyIntolerance.readAllergies via patient-scoped search only
  • system/DocumentReference.readSearch, normalized to metadata only. No narrative content, no Binary follow-up.
No Binary. Eligibility remains deterministic on structured fields; Epic registration omits Binary web services and any Binary read scope.
04_CLINICAL_DATA_BOUNDARY

Clinical data boundary

  • Evidence cohorts (Epic cohorts) are defined by approved protocol logic or institutional configuration.
  • PHI is processed inside the institution-approved deployment boundary, with tenant isolation per the deploy contract.
  • PHI never reaches an LLM. Criterion PASS / REVIEW / FAIL comes only from the deterministic engine on ingested FHIR. No patient identifiers or cohort-linked fields are serialized into any optional external request path.
  • Helm ships deny-by-default API egress NetworkPolicy with explicit CIDR allowlists for Epic FHIR, OIDC, in-cluster Postgres/Redis, and DNS.
  • Screening outputs are criterion-level PASS, REVIEW, or FAIL with evidence references, never an ungrounded generative verdict.
05_AUDIT_AND_REPLAY

Audit, determinism, and replay

  • Decisions are reproducible for a fixed protocol version, cohort snapshot, evidence snapshot, and evaluation_as_of.
  • Every patient outcome is recorded at criterion level with PASS, FAIL, or REVIEW and a rationale.
  • Protocol amendments link to versioned protocol IDs and engine versions.
  • Replay classifies agreement or divergence. No opaque "model decided" eligibility.
07_SECURITY_AND_PRIVACY

Security, privacy, and logging

  • Signing keys and Epic access tokens are server-side secrets; FHIR and browser traffic use TLS.
  • Structured logs favor stable identifiers. Field-level exclusion lists omit free-text clinical narrative from default log templates.
  • Data at rest encryption and retention follow the deployment architecture and the customer BAA / DPA.
